# Malloc is not a Magic

대부분의 언어는 메모리를 OS(운영체제)로부터 할당받을 때 `new` 키워드를 사용하며, 대개 `GC(Garbage Collector)`가 내장되어 있어 개발자가 수동 해제를 고려하지 않는다. 

반면 C/C++은 `malloc`과 `free`, `new`와 `delete`를 사용해 메모리를 직접 관리할 수 있다. 이러한 수동 메모리 관리 이점 덕분에 C/C++은 퍼포먼스 측면에서 다른 언어들보다 압도적인 우위를 점한다. 실제로 가상 함수 오버헤드, 메모리 레이아웃 제어, 캐시 효율 등을 직접 테스트하며 최적화해 본 결과, 작은 부분들이 모여 2.x 배 이상의 성능 향상이 누적되는 것을 경험했다.

여기서 핵심은 'OS와 통신하는 비용(System Call)이 가장 크다' 는 점이었고, 성능 향상을 위한 최적화 기법들 역시 결국 이 비용을 줄이기 위해 고안된 것이었다. 그렇다면 '도대체 OS로부터 메모리를 빌려오는 과정은 어떻게 이루어지는가?' 에 대한 근본적인 의문에서 가볍게 정리해보았다.

`main.c`에는 아주 단순한 형태의 allocator가 들어 있다.

- `mmap`으로 OS에 메모리 페이지를 요청한다.
- `Block` 헤더에 메타데이터를 저장한다.
- 비어 있는 블록이 있으면 재사용한다.
- 큰 free 블록은 필요한 크기만큼 분할한다.
- `free` 시 인접한 free 블록을 다시 병합한다.
- 힙 상태를 출력해 allocation 흐름을 눈으로 확인할 수 있게 했다.

## 메타정보 관리

빌려온 메모리를 관리하기 위해서, double linked list 형태로 `Block` 구조체를 정의했다.   
실제 사용할 때는 `Block` 헤더 바로 뒤의 메모리 영역(size)을 반환한다.

```c
typedef struct Block {
  size_t size;
  bool is_free;
  struct Block *prev;
  struct Block *next;
} Block;
```

## 할당 흐름

`my_malloc(size)`의 흐름은 다음과 같다.

1. 먼저 재사용 가능한 free 블록이 있는지 리스트를 순회한다.
2. 충분히 큰 블록이 있으면 분할해서 재사용한다.
3. 없다면 `mmap`으로 OS에 한 페이지 이상을 요청한다.
4. 새 메모리 영역을 리스트에 연결하고, 분할 후 payload 영역을 반환한다.

## 해제 흐름

`my_free(ptr)`의 흐름은 다음과 같다.

1. 해당 블록을 free 상태로 바꾼다(단순 마킹).
2. 다음 블록이 free이고 실제로 메모리상 인접해 있으면 병합한다.
3. 이전 블록이 free이고 실제로 메모리상 인접해 있으면 다시 병합한다.

> 여기서 병합의 조건은 실제 메모리 주소가 인접해야하는 점인데, 이게 heap 이 stack 보다 느린 이유이기도 하다(접근 비용). heap 영역의 메모리를 요청할 떄는 페이지단위(4KB) 그리고 물리적인 위치도 랜덤이다.

## 나가며

목표는 표준 라이브러리의 `malloc`을 그대로 재구현하는 것이 아니었다.   
목표는 메모리를 더 이상 블랙박스로 두지 않는 것이었다.   
이 레벨까지 내려와 보면 평소 추상적으로만 느껴지던 주제들이 훨씬 구체적으로 연결되기 시작한다.

- 왜 heap allocation은 공짜가 아닌가
- 왜 객체의 메모리 레이아웃이 중요한가
- 왜 간접 참조가 캐시 효율에 영향을 주는가
- 왜 커스텀 allocator와 메모리 풀이 필요한가
- 왜 C/C++는 다른 관리형 언어보다 성능 특성을 더 직접적으로 드러낼 수 있는가
